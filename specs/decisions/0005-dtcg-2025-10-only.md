# ADR 0005 — Pin to DTCG · 2025.10 as the only supported format

- **Status:** accepted
- **Date:** 2026-06-23
- **Deciders:** PO
- **Related:** [../constitution.md](../constitution.md) §1 (non-goals),
  §3.2 (spec conformance),
  [../requirements/req-0001.md](../requirements/req-0001.md)

## Context

The requirement names a specific snapshot of the W3C DTCG draft:
"Design Tokens Format Module 2025.10" (as referenced at
<https://www.designtokens.org/tr/drafts/format/>). The format is still
a draft and continues to evolve. There is also a sizeable installed
base of files in the **older Tokens Studio format**, which predates
DTCG and uses different conventions (`type` vs `$type`, no `$value`
wrapping, different alias syntax).

Three positions:

- **Support both DTCG and Tokens Studio** to maximise reach.
- **Support DTCG only, tracking the latest draft.**
- **Support DTCG, pinned to the 2025.10 snapshot, with explicit
  deprecation handling for newer drafts.**

## Decision

The plugin supports W3C DTCG · 2025.10 only. Tokens Studio format files
are out of scope (constitution §1 non-goal).

When a future DTCG draft introduces a breaking change, the response is:

1. Add a new fixture under `tests/fixtures/` for the new draft.
2. Decide per-change whether to bump the supported version (breaking)
   or to detect and translate (additive).
3. Surface unsupported `$type` values in the import console as warnings,
   never silently. Log them in `docs/spec-deviations.md` per
   constitution §3.2.

The parser today already enforces this: unsupported `$type` values
produce a warning and skip the token rather than crashing
([../../plugin/src/shared/dtcg/parse.ts](../../plugin/src/shared/dtcg/parse.ts)).

## Consequences

**Positive**

- Scope is bounded. We don't have to maintain two parsers or reason
  about which dialect a file belongs to.
- Deprecation policy is explicit; users running newer-draft files get
  a per-token warning, not a silent drop.

**Negative**

- Users with Tokens Studio archives must convert before importing.
  There are existing CLIs for this conversion; we link to one rather
  than building it in.
- Pinning to a draft means the supported version will eventually be
  "old". The deprecation process above is the mitigation.
